# Festo OAuth Backend Server

A secure Express.js OAuth backend server for handling PingOne authentication and token management. This server bridges your Expo mobile application and PingOne's OAuth 2.0 service.

## Features

- 🔐 **Secure OAuth Token Exchange** - Safely exchange authorization codes for tokens
- 🔄 **Token Refresh** - Refresh expired access tokens
- 👤 **User Info Retrieval** - Fetch detailed user information from PingOne
- 🛑 **Token Revocation** - Securely revoke tokens on logout
- 🌐 **CORS Support** - Handle cross-origin requests from both development and production Expo apps
- 🚀 **Production Ready** - Built with security best practices
- 📊 **Error Handling** - Comprehensive error responses for debugging

## Architecture

```
┌──────────────────┐
│  Mobile App      │
│  (Expo/React)    │
└────────┬─────────┘
         │
         │ 1. Request authorization code
         │ (with Expo AuthSession)
         ▼
┌──────────────────┐
│  PingOne OAuth   │
│  Authorization   │
└────────┬─────────┘
         │
         │ 2. Return auth code via redirect
         │
         ▼
┌──────────────────────┐
│  Backend Server      │
│  (This service)      │
└────────┬─────────────┘
         │
         │ 3. Exchange code for tokens
         │    (secure, client_secret not exposed)
         │
         ▼
┌──────────────────┐
│  PingOne API     │
└────────────────┘
```

## Prerequisites

- Node.js 14+ and npm
- PingOne account with configured OAuth application
- Environment variables configured

## Installation

### 1. Install Dependencies

```bash
cd backend
npm install
```

### 2. Configure Environment Variables

Create `.env` file:

```env
# PingOne Configuration
PINGONE_ENV_ID=your_environment_id
PINGONE_CLIENT_ID=your_client_id
PINGONE_CLIENT_SECRET=your_client_secret

# Server Configuration
PORT=3001
NODE_ENV=development

# Frontend URLs
FRONTEND_URL=http://localhost:19000
```

## Getting Credentials

### From PingOne Admin Console

1. Sign in to your PingOne admin dashboard
2. Go to **Connections > Applications** (or **Management > Applications**)
3. Create a new **Web** or **Native** application
4. Under **Credentials**, copy:
   - **Client ID**
   - **Client Secret**
5. Make note of your **Environment ID** (visible in dashboard URL or settings)

### Redirect URI Configuration

In your PingOne application settings, add redirect URIs:

**Development:**

```
http://localhost:3001/callback
festo-mobile://callback  # generated by Expo AuthSession with custom scheme
```

> **Tip:** the mobile app generates the redirect URI dynamically using
> `AuthSession.makeRedirectUri` with a custom URL scheme. It typically matches
> the value above (`festo-mobile://callback`), but checking the app logs can
> verify the exact string if you prefer.

**Production:**

```
https://yourdomain.com/callback
exp://your-app-slug.exp.direct/--/pingone-callback
```

## Running the Server

### Development Mode (with auto-reload)



## API Endpoints

### 1. Exchange Authorization Code for Tokens

**Request:**

```bash
POST /exchange-code
Content-Type: application/json

{
  "code": "authorization_code_from_pingone",
  "redirectUrl": "festo-mobile://callback"
}
```

**Response:**

```json
{
  "accessToken": "access_token_xxx",
  "refreshToken": "refresh_token_xxx",
  "idToken": "id_token_xxx",
  "expiresIn": 3600,
  "user": {
    "id": "user-id",
    "email": "user@example.com",
    "name": "John Doe",
    "givenName": "John",
    "familyName": "Doe",
    "picture": "https://...",
    "locale": "en-US"
  }
}
```

**Purpose:** This is called by your mobile app after user logs in with PingOne. It securely exchanges the temporary authorization code for access and refresh tokens. The client_secret never leaves the backend server.

### 2. Refresh Access Token

**Request:**

```bash
POST /refresh-token
Content-Type: application/json

{
  "refreshToken": "refresh_token_xxx"
}
```

**Response:**

```json
{
  "accessToken": "new_access_token_xxx",
  "refreshToken": "refresh_token_xxx",
  "expiresIn": 3600
}
```

**Purpose:** Called when the access token expires. Doesn't require re-authentication.

### 3. Revoke Token

**Request:**

```bash
POST /revoke
Content-Type: application/json

{
  "token": "access_token_to_revoke"
}
```

**Response:**

```json
{
  "message": "Token revoked successfully"
}
```

**Purpose:** Called on logout to immediately invalidate the token.

### 4. Get User Information

**Request:**

```bash
GET /user-info
Authorization: Bearer access_token_xxx
```

**Response:**

```json
{
  "id": "user-id",
  "email": "user@example.com",
  "name": "John Doe",
  "givenName": "John",
  "familyName": "Doe",
  "createdAt": "2024-03-06T10:30:00Z",
  "updatedAt": "2024-03-06T10:30:00Z"
}
```

**Purpose:** Fetch additional user details from PingOne using a valid access token.

### 5. Health Check

**Request:**

```bash
GET /health
```

**Response:**

```json
{
  "status": "OK",
  "message": "OAuth backend server is running"
}
```

**Purpose:** Check if backend server is running and healthy.

## Error Responses

All error responses follow this format:

```json
{
  "error": "error_description",
  "details": {
    "error": "PingOne_error_code",
    "error_description": "Detailed error message from PingOne"
  }
}
```

### Common Errors

| Status | Error | Cause |
|--------|-------|-------|
| 400 | Missing authorization code | Code wasn't provided in request |
| 400 | Missing refresh token | Refresh token wasn't provided |
| 401 | Invalid token | Token format or signature invalid |
| 401 | Invalid credentials | Client ID/Secret incorrect |
| 500 | Invalid environment | Environment ID doesn't exist |
| 500 | Internal server error | Server-side issue |

## Security Considerations

### 1. **Client Secret Security**

- ✅ **DO** keep client_secret only on backend
- ❌ **DON'T** expose client_secret in frontend code
- ❌ **DON'T** commit .env files to version control

### 2. **HTTPS in Production**

- Use HTTPS (TLS) for all production endpoints
- PingOne will enforce HTTPS for production apps

### 3. **Token Storage**

- Mobile app stores tokens in Secure Store (encrypted)
- Backend never stores tokens persistently
- Tokens cached only during request lifecycle

### 4. **Token Validation**

- Backend validates tokens with PingOne
- Refresh tokens have limited lifetime
- Access tokens expire and must be refreshed

### 5. **CORS Protection**

- Server validates origin headers
- Only whitelisted origins can access endpoints
- Production origins should be explicitly configured

### 6. **Rate Limiting** (Recommended)

Add rate limiting for production:

```bash
npm install express-rate-limit
```

```javascript
const rateLimit = require('express-rate-limit');
const limiter = rateLimit({
  windowMs: 15 * 60 * 1000, // 15 minutes
  max: 100 // limit each IP to 100 requests per windowMs
});

app.use('/exchange-code', limiter);
app.use('/refresh-token', limiter);
```

## Troubleshooting

### port 3001: Address already in use

```bash
# Find process using port
lsof -i :3001

# Kill process
kill -9 <PID>

# Or use different port
PORT=3002 npm start
```

### "invalid_client" Error

- Check Client ID and Client Secret are correct
- Verify environment ID matches

### "invalid_grant" Error

- Authorization code may have expired
- Code was already used (can only use once)

### "invalid_request_uri" Error

- Redirect URI doesn't match PingOne configuration
- Check exact spelling and protocol

### CORS Errors from Mobile

- Check `cors` configuration in `index.js`
- Verify mobile app origin is whitelisted
- Ensure backend is accessible from device network
